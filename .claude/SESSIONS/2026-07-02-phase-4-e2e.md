## Réalisé (Étape B)
### BD de test jetable, sans jamais toucher `aio`
- PostgreSQL jetable via **Docker** (`postgres:16`, port **5434**, base `aio_e2e`).
- Garde-fou `test/e2e-db/setup-env.ts` (jest `setupFiles`) : **exige** `TEST_DATABASE_URL`,
  **refuse** explicitement une URL pointant sur `aio`, puis copie l'URL de test dans
  `DATABASE_URL`/`DIRECT_URL` avant tout import applicatif.
- Migrations appliquées sur la base de test via `prisma migrate deploy` (88, dont les 2 Phase 4).

### 3 garanties prouvées contre un vrai PostgreSQL (`test/e2e-db/*.db-e2e-spec.ts`)
1. **Isolation** (`isolation.db-e2e-spec.ts`) — `VenteValidationService.validerPlat` : une variante
   dont le plat appartient au restaurant B est **acceptée pour B** et **refusée pour A**
   (`PLAT_NOT_FOUND`), sans mutation de stock. Exerce le correctif Prisma 5.7 sur vraie base.
2. **Atomicité** (`atomicity.db-e2e-spec.ts`) — `PosSessionRestaurantService.ouvrirSession` :
   panne déterministe injectée sur la 2ᵉ étape (occupation de table) via un proxy sur le client
   transactionnel **réel** ; transaction + création + rollback 100 % réels →
   **aucune `PosSession` persistée**, table restée `LIBRE`. Contrôle positif : sans panne, session
   créée + table `OCCUPEE`.
3. **Audit** (`audit.db-e2e-spec.ts`) — seam `AuditJournalService.record(tx, …)` (celui qu'utilisent
   tous les chemins sensibles) exercé dans une vraie `$transaction` : écrit bien une ligne
   `historique` immuable (acteur/entité/portée/avant-après) ; et si l'action échoue, la trace est
   **annulée avec la transaction** (pas de trace fantôme).

### Commande unique reproductible
- **`pnpm test:e2e:db`** (`scripts/e2e-db.mjs`) : démarre le postgres jetable → migrate →
  jest (config e2e-BD) → **détruit le conteneur dans tous les cas** (try/finally).
- Résultat : **3 suites / 7 tests verts**. `aio` jamais touchée.
